3. KEY DECISIONS AND WHY
-------------------------
(a) Used @next/third-parties <GoogleAnalytics> rather than pasting the raw
    gtag snippet. Reason: it loads gtag via next/script so it is scheduled
    rather than render-blocking.

    !! CORRECTION MADE THIS SESSION — DO NOT REPEAT THE ORIGINAL CLAIM !!
    I initially told the founder the package handles App Router route changes.
    That is FALSE. Reading node_modules/@next/third-parties/dist/google/ga.js,
    the component only injects the gtag init snippet + the gtag/js tag. There
    is NO router subscription.

    SPA pageviews work because GA4 **Enhanced Measurement** ("page changes
    based on browser history events") is ON BY DEFAULT in the property.
    If anyone disables Enhanced Measurement in the GA4 admin, every pageview
    after the first hard load stops silently, and Analytics.tsx would then
    need to send page_view itself on route change.

(b) Consent is enforced by NOT MOUNTING <GoogleAnalytics> until the visitor
    accepts — deliberately NOT Google Consent Mode. Reason: no cookie is
    written and no request leaves the browser pre-consent, which is a far
    easier POPIA position to defend than "we sent the request but flagged it
    as unconsented."

(c) Consent state is read with useSyncExternalStore, NOT useState+useEffect.
    Reason: this repo's lint rule `react-hooks/set-state-in-effect` rejects
    setState in an effect body. Details:
      - localStorage key: `gg-cookie-consent`, values "granted" | "denied"
      - server snapshot is a distinct "unread" state so the banner is not
        baked into SSR HTML and flashed at people who already answered
      - a module-level `fallback` var carries the answer when localStorage is
        blocked (private mode); without it the write fails silently and the
        banner could never be dismissed
      - custom `gg-cookie-consent-change` event, because the native `storage`
        event only fires in OTHER tabs

(d) Banner does not render at all when gaId is unset. Reason: asking consent
    for something that cannot load is theatre and trains people to dismiss a
    banner that does nothing. Side effect: the banner appearing IS the
    confirmation that the env var took effect.

(e) Banner styling matches the house system: `rounded-block` (40px) + `grain`,
    bark background, peach text, ember Accept. A peach hairline border was
    added because on mobile the bark banner sat flush against the ember hero
    band and read as one continuous dark shape.

5. OPEN ITEM CARRIED FORWARD — FOCUS-RING CONTRAST AUDIT
----------------------------------------------------------
This is the agreed NEXT PIECE OF WORK. It is on the punch list in
GoodGround_memory.md and recorded as trap #13.

Problem: app/globals.css sets a global `:focus-visible` outline of 2px solid
var(--color-ember). Measured contrast:
    ember #ac4119 on cream  = 4.90:1  PASS
    ember #ac4119 on bark #331816 = 2.75:1  FAIL (WCAG needs 3:1 for focus
                                                  indicators)

Fixed so far: the cookie banner only, via `[&_:focus-visible]:outline-peach`
(peach #fde5c1 on bark = 13.4:1).

STILL TO AUDIT — any interactive element sitting on a bark or pine surface:
    components/sections/Process.tsx
    components/sections/CTABanner.tsx
    components/Marquee.tsx
    components/Footer.tsx  (footer links)
    components/Nav.tsx     (nav is pine #1c261d — check this one too)
    components/sections/WhatWeBuild.tsx
    components/sections/Hero.tsx (ember band)
Plus anything using components/Block.tsx with a dark variant.

Suggested approach: rather than sprinkling the override per component,
consider scoping it once — e.g. a utility applied by Block.tsx for dark
variants, so future dark panels inherit the correct ring automatically.
Verify with real measurement, not eyeballing.

TRAP WHEN MEASURING CONTRAST IN-BROWSER: read alpha from a TRANSPARENT canvas.
Compositing over white first makes every colour read opaque and produces false
failures. (Already recorded in the memory doc.)
